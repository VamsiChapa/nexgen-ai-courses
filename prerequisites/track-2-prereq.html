Day P2

Mathematics for Data Science

Machine learning is applied mathematics. You don't need to be a mathematician, but you do need to understand what the maths is doing — at a conceptual level. Today builds that foundation.

Statistics Fundamentals

Descriptive Statistics

Before building models, you need to describe and understand your data. These are the tools:

  • Mean — average value. Sensitive to outliers. If your dataset is [1, 2, 3, 4, 100], the mean is 22 — not representative of most values.
  • Median — the middle value when sorted. Robust to outliers. For the same dataset: median is 3.
  • Mode — the most frequent value. Useful for categorical data.
  • Variance — average of squared differences from the mean. Measures how spread out values are.
  • Standard Deviation — square root of variance. Same unit as your data. A low std dev means data is clustered near the mean; high means it's spread out.
  • Percentiles — the p-th percentile is the value below which p% of data falls. The 50th percentile is the median. Used in ML to understand data distribution.

Practical rule: always look at mean, median, and std dev together. If mean and median are very different, your data likely has outliers or a skewed distribution — which will affect your model.

Probability Basics

Probability is the language of uncertainty, and ML models output probabilities constantly. Core concepts:

  • Probability P(A) — a number between 0 and 1 expressing how likely event A is. P=0 means impossible, P=1 means certain.
  • Conditional Probability P(A|B) — probability of A given that B has occurred. A spam filter computes P(spam | "buy now" in email).
  • Independence — A and B are independent if knowing B tells you nothing about A. This assumption underlies Naive Bayes classifiers.
Linear Algebra Intuition

Vectors and Matrices

Virtually all ML operations reduce to linear algebra. A vector is just a list of numbers (a row or column). A matrix is a 2D grid of numbers — like a spreadsheet.

In ML: a dataset with 1000 rows and 20 features is a matrix of shape (1000, 20). Each row is a data point. Each column is a feature. Your model parameters are also vectors and matrices.

The dot product of two vectors multiplies corresponding elements and sums them. It measures how "aligned" two vectors are — core to neural network forward passes.

Python — Linear Algebra with NumPy
import numpy as np a = np.array([1, 2, 3]) b = np.array([4, 5, 6]) # Dot product: 1*4 + 2*5 + 3*6 = 4+10+18 = 32 print(np.dot(a, b)) # 32 # Matrix multiplication A = np.array([[1, 2], [3, 4]]) B = np.array([[5, 6], [7, 8]]) print(A @ B) # [[19, 22], [43, 50]] # Statistics with NumPy data = np.array([4, 7, 13, 2, 1, 8]) print(f"Mean: {np.mean(data):.2f}") print(f"Std: {np.std(data):.2f}") print(f"Median: {np.median(data):.2f}")
Calculus Intuition

Derivatives = Slope = Rate of Change

You don't need to calculate derivatives by hand. You need to understand what a derivative means: the derivative of a function at a point tells you the slope of the function there — how fast it's changing, and in which direction.

This matters because gradient descent — the algorithm that trains virtually all ML models — uses derivatives to find the direction in which to adjust model parameters in order to reduce error. The derivative tells the model "if you increase this parameter slightly, does the error go up or down?"

Key intuition: if the derivative (slope) is positive, the function is going up — decrease the parameter to lower error. If it's negative, the function is going down — increase the parameter. Gradient descent follows the slope downhill.

Day P4

Git & GitHub Basics

Version control is non-negotiable for any serious developer. Git tracks changes to your code over time. GitHub hosts your code online and becomes your professional portfolio.

Why Version Control Matters

Without version control, you'd make a change, break something, and have no way to get back to the working version. With Git, every state of your code is saved — you can jump back to any point in history, experiment on branches without affecting your main code, and collaborate with others without overwriting each other's work.

Real scenario: you're training a model, change the architecture, and performance drops. With Git, you can run git checkout previous-commit and instantly restore the version that worked. This is not optional — this is how professionals work.

Core Git Concepts

The Mental Model

  • Repository (repo) — a folder tracked by Git. Contains all your files and their complete history.
  • Commit — a snapshot of your files at a specific point in time. Each commit has a message describing what changed and why.
  • Branch — an independent line of development. The default branch is main. Create a branch to work on a feature without touching main.
  • Merge — combining changes from one branch into another. When your feature is done, you merge it back to main.
  • Remote — a version of your repo hosted online (e.g., on GitHub). You push local commits to remote, pull remote changes to local.

Essential Commands

Terminal — Git Commands
# Start tracking a folder git init # Check what's changed git status # Stage files for commit (. means all files) git add . git add specific-file.py # or add just one file # Save a snapshot with a descriptive message git commit -m "Add data loading and preprocessing pipeline" # Connect to GitHub and push git remote add origin https://github.com/yourusername/repo-name.git git push -u origin main # Get latest changes from GitHub git pull # See full commit history git log --oneline # Create and switch to a new branch git checkout -b feature/new-model-architecture

Writing Good Commit Messages

A commit message should complete the sentence "This commit will...". Bad: fixed stuff. Good: Fix null handling in data preprocessing pipeline. Your future self will thank you.

  • Keep the first line under 72 characters
  • Use imperative mood: "Add", "Fix", "Update", "Remove" — not "Added" or "Fixed"
  • Reference what changed and why, not what the code does

Day P5

Working with Data Files

Data science starts with data — and data lives in files. Today you master the most common data file formats and learn to read, convert, and write them in Python.

Data File Formats

CSV — Comma-Separated Values

The most universal format. A plain text file where each line is a row and columns are separated by commas. Every spreadsheet application and data tool can read CSV. Use it for: small-to-medium datasets that need to be human-readable or shared across tools.

Gotcha: CSV doesn't store data types. Every value is read as a string by default. You must explicitly convert columns to numbers, dates, or booleans when loading.

JSON — JavaScript Object Notation

A text format for structured, nested data. Think dictionaries and lists. APIs almost always return JSON. Use it for: API responses, nested/hierarchical data, configuration files.

JSON can express lists within lists, dictionaries within dictionaries — things CSV can't represent without flattening. This makes it more flexible but harder to query as a table.

Parquet — Columnar Storage for Big Data

Parquet is a binary format designed for performance. Unlike CSV (row-oriented), Parquet stores data column by column. If your query only needs 3 columns out of 100, it only reads those 3. This makes it 10-100x faster for analytical queries on large datasets.

Parquet also stores data types natively, compresses extremely well, and is the standard format in Spark, Databricks, and cloud data warehouses. You'll see it frequently in production data pipelines.

Reading Files in Python
Python — Reading All Three Formats
import pandas as pd import json # ── CSV ────────────────────────────────────────────────── df_csv = pd.read_csv('data.csv') # Force data types on load df_csv = pd.read_csv('data.csv', dtype={'age': int, 'name': str}) # Handle encoding issues (common with non-English data) df_csv = pd.read_csv('data.csv', encoding='utf-8') # ── JSON ───────────────────────────────────────────────── with open('data.json') as f: data = json.load(f) # Python dict/list df_json = pd.read_json('data.json') # for flat JSON arrays # ── Parquet ────────────────────────────────────────────── # pip install pyarrow or fastparquet first df_parquet = pd.read_parquet('data.parquet') # ── Conversion ─────────────────────────────────────────── df_csv.to_json('output.json', orient='records') df_csv.to_parquet('output.parquet', compression='snappy') # ── Chunked reading for large files ────────────────────── for chunk in pd.read_csv('large_file.csv', chunksize=10000): process(chunk) # process 10,000 rows at a time

Day P6

SQL Basics for Data Scientists

SQL is not just for backend developers. Data scientists use it constantly — to pull training data, explore databases, and join datasets. Today you learn enough SQL to be effective.

Why SQL Matters for ML

Your training data lives in databases. When you need to pull 500,000 rows for a model, filtering and aggregating in SQL before loading is orders of magnitude faster than loading everything into pandas and filtering in Python. SQL also runs close to the data — no network transfer of unused rows.

Additionally, many ML platforms (BigQuery ML, Databricks, Snowflake) let you train models directly with SQL. Knowing SQL opens doors that Python alone doesn't.

Core SQL Syntax
SQL — Essential Queries
-- Basic SELECT SELECT name, age, score FROM students WHERE score > 80 ORDER BY score DESC LIMIT 10; -- Aggregation SELECT grade, COUNT(*) AS student_count, AVG(score) AS avg_score, MAX(score) AS top_score FROM students GROUP BY grade HAVING AVG(score) > 75; -- JOIN: combine two tables SELECT s.name, s.score, c.course_name FROM students s JOIN courses c ON s.course_id = c.id WHERE c.category = 'AI';

SQL in Python

Python — SQL with sqlite3 and Pandas
import sqlite3 import pandas as pd # Connect to a local SQLite database (creates it if missing) conn = sqlite3.connect('students.db') # Write a DataFrame to the database df.to_sql('students', conn, if_exists='replace', index=False) # Read query results directly into a DataFrame query = """ SELECT grade, AVG(score) as avg_score FROM students GROUP BY grade ORDER BY avg_score DESC """ results = pd.read_sql_query(query, conn) print(results) conn.close()
When SQL Beats Pandas
  • Data is in a database already — pulling with SQL is far faster than loading to CSV then pandas
  • Dataset is too large for RAM — SQL can query terabytes; pandas loads everything into memory
  • Multiple people need the same data — SQL queries are shareable, reproducible, and version-controllable
  • You need JOINs across many tables — SQL's join syntax is cleaner and more performant than pandas merges for complex joins
